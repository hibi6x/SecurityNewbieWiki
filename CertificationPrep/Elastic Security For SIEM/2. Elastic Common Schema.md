# Elastic Common Schema (ECS) — Making Sense of Inconsistent Security Data

## The Problem: Inconsistent Security Data

Security data comes from **many different sources**, each recording information in its own format.

### Common Security Data Sources

| Source | What It Captures |
|---|---|
| **Network Logs** | Connection attempts, traffic patterns (routers, switches, web/DNS/DHCP servers) |
| **Endpoints** | OS & app activity, application behavior, audit events |
| **Security Devices** | Alerts & telemetry from IDS/IPS, firewalls, antivirus |
| **Cloud Services** | Audit trails — access control, user activity, resource usage, config changes (AWS, Azure, GCP) |

---

## Why Inconsistency Is a Problem

Different systems use **different field names for the same concept**.

**Example — Source IP field across systems:**

| System | Field Name |
|---|---|
| Windows Security Log | `IP address` |
| AWS CloudTrail | `source IP address` |
| Suricata Alert | `src_ip` |

> All three refer to the same thing — the originating IP — but Elastic treats them as **completely different fields**.

### Consequences of Inconsistent Data
- **Search** — You must remember every field name variation across every source (time-consuming & error-prone)
- **Dashboards** — Lose context without a shared structure
- **Detections** — Can miss key signals when fields don't align

---

## The Solution: Data Normalization

**Data normalization** is the process of converting information from many different formats into a **consistent structure**, without changing the meaning of the data.

- Makes data **predictable** and **uniform**
- Makes data **much easier to analyze**

---

## Elastic Common Schema (ECS)

**ECS** is an **open-source specification** that defines a **common set of fields** for storing event data in Elasticsearch.

### Key Properties
- Enables **consistent normalization, analysis, and correlation** across diverse data sources
- Normalizes data from different sources into a **single schema**
- Reduces complexity during **correlation and detection**

> 💡 In simple terms: **ECS makes all your security data speak the same language.**

---

## Summary

```
Many data sources → Different field names & formats
            ↓
    Data Normalization
            ↓
  Elastic Common Schema (ECS)
            ↓
  Single, unified structure → easier search, visualization & detection
```

# How ECS Normalizes and Unifies Security Data

## Normalization vs. ECS — Key Distinction

| Concept | Definition |
|---|---|
| **Data Normalization** | A general process of making data consistent across formats |
| **Elastic Common Schema (ECS)** | The Elastic-specific standard that *defines* how that consistency is structured |

---

## How ECS Works

ECS applies normalization by **mapping different field names and formats** from various sources into a **single, predictable schema**.

- All events — regardless of origin — use the **same field names and data types**
- The result is **clean, consistent, and searchable** security data

**Example mapping:**

| Original Field (source) | ECS Normalized Field |
|---|---|
| `IP address` (Windows) | `source.ip` |
| `source IP address` (AWS CloudTrail) | `source.ip` |
| `src_ip` (Suricata) | `source.ip` |

---

## How ECS-Compliant Data Is Delivered

### 1. Elastic Agent Integrations & Beats
- Ship **ECS-compliant data out-of-the-box**
- Logs arrive **already aligned** to the correct ECS fields — no extra manual work required

### 2. Automatic Import
- A feature in **Elastic Security** for data sources that **don't have pre-built integrations**
- Uses a **large language model (LLM)** to:
  1. Analyze your sample data
  2. Build a **custom integration** that normalizes your data to ECS
- Enables Elastic Security to immediately **correlate, visualize, and detect** patterns — even across completely different original schemas

---

## Benefits of ECS Normalization

### 1. 🔗 Seamless Data Correlation
- Standardized field names allow Elastic Security to **link related events** across logs, metrics, and threat intelligence
- Provides a **unified view** of activity instead of fragmented data silos

### 2. 🎯 Accurate Detections & Triage
- Pre-built and custom **detection rules rely on ECS fields** for context
- Key fields used in rules: `source.ip`, `event.category`, `process.name`
- Consistent schema means **alerts trigger reliably** and investigations have the full context needed

### 3. ♻️ Reusable Content
- Dashboards, alerts, and analytics built on ECS **work across cloud, endpoint, and network sources**
- All sources follow the same structure, so content is universally applicable

### 4. 🔮 Future-Proofing
- New Elastic features and integrations **automatically work with ECS-normalized data**
- Workflows stay consistent **as your environment grows and evolves**

# Inside ECS — The Fields That Power Elastic Security

## Required ECS Fields (Core Classification)

These fields are **mandatory** — without them, Elastic Security **cannot classify events** or use them in rules, dashboards, or timelines.

| Field | Purpose |
|---|---|
| `@timestamp` | Records **when** the event occurred |
| `ecs.version` | Identifies the **ECS version** in use |
| `event.kind` | Defines the **nature** of the event (e.g., alert, event, signal) |
| `event.category` | **Classifies** the type of activity (e.g., network, process, file) |
| `event.type` | Further **sub-classifies** the event for analysis |

> ⚠️ If these fields are missing or inconsistent, key security features **will not work as expected**.

---

## Host & Process Fields — Endpoint Visibility

Critical for understanding **what happened on which machine**.

| Field | Purpose |
|---|---|
| `host.name` | Identifies the **system** where activity occurred |
| `process.name` | Identifies **which process** ran |
| `process.pid` | Identifies the **specific process instance** |

### What These Enable
- Identifying **uncommon or suspicious processes**
- Tracking **parent-child process relationships**
- Building **visual process trees**

> Without standardized host/process fields, analysts **cannot reliably trace activity** back to a specific machine or process during an investigation.

---

## Network Fields — Traffic Visibility

Network data comes from many sources: traffic logs, firewalls, proxies, and cloud platforms.

| Field | Purpose |
|---|---|
| `source.ip` | Shows **where** network traffic originated |
| `destination.ip` | Shows **where** traffic was sent |
| `geo.*` fields | Adds **geographic context** based on IP address |

### What These Enable
- **Map traffic** across the environment
- **Detect anomalies** in connections
- **Correlate sessions** across different network tools
- Analyze network activity as part of a **larger security story**, not in isolation

---

## Protocol Fields — Web & Encrypted Traffic Context

Higher-level protocol fields provide critical context for deeper traffic analysis.

| Protocol | Key Fields |
|---|---|
| **DNS** | Query names, response types |
| **HTTP** | Methods, URIs, status codes, user agents |
| **TLS** | Fingerprints, certificate details |

### What These Enable
- Detecting **suspicious domains**
- Identifying **unusual HTTP behavior**
- Flagging **abnormal TLS fingerprints**

> Without standardized protocol fields, this context would be **fragmented or lost entirely**.

---

## How ECS Fields Power Core Security Functions

Everything in Elastic Security depends on **consistent field names**:

### 🔗 Correlation
- ECS standardizes fields **across vendors and data sources**
- Elastic Security can connect events into a **single, meaningful timeline**

### 🎯 Detection
- Pre-built rules are **written using ECS fields**
- When data follows the schema, **alerts trigger consistently across all sources**

### 🔍 Investigation
- ECS **eliminates guesswork** — analysts can move freely between alerts, timelines, and cases
- No need to reinterpret data per source
- Leads to **faster, more reliable investigations** and **better team collaboration**
